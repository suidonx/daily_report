- どうやって品質保証をする？

		- バグ発見の可能性が高まるため、テストケースを多くした方がいい
		- 例えば、商品レビューの数が多いほど安心できるよね
- じゃあ無限テストケース用意すれば良くない？
	- 現実的に全ての組み合わせを用意するのは無理
	- 完璧なテストはできないという前提を持つこと。
- ソフトウェアテストの種類
	- ユニットテスト

	- 統合テスト
		- メソッドの組み合わせのテスト
	- UI/システムテスト
		データベースやUI部品を組み合わせたテスト

### ソフトウェアテスト技法
- なるべく少ないテストケースでなるべく多くのバグを発見するための技術
- テストケースは多ければ多いほど意味があるものではない。
	- ちゃんと組み合わせを網羅させる必要がある
- 同値クラステスト
    - 同値クラスとはテスト的に同じ意味になる値
- テスト的に同じ意味とは、a=2, a=3はif文に入るからそれぞれ同じ意味になる。
- 同値クラスは条件に1個あればいいので最適化できる。

こういう条件分岐のテストケース
if, elif, elseに進むかテストするため、2, 3, 2と3以外の値の3つ必要
```
if a == 2:
	pass
elif a == 3:
	pass
else:
	pass
```

- 境界値テスト
	- 不等式を使う条件分岐のテストケース
	- 不等号に=が入っている、入っていないのミスなど境界値付近にはバグが潜みやすい。
	- a = 2, 3, 4, 9, 10, 11,が最適になる。
```
if a <= 3:
	pass
elif a >= 10:
	pass
else:
	pass
```

テストケース設計に必要なのは？
- 現実的に実行できるテストケースまで量を削減する
- 少ないテストケースでもより多くのバグを発見すること

テスト技法を学ぶには、この両取りを意識していく
- テストケースを減らすこと
- 網羅性を担保すること

### テスト技法の基礎知識
- 網羅性
	- テストの抜け漏れをなくす

- ピンポイント
	- 怪しいところを入念に調べる

これを使い分けていく

テスト設計アプローチ
どのようにテストを設計していくか
- 契約によるテスト
	- 事前条件によるテストすること
		- メソッド(モジュール)が正しく呼び出される際に満たすべき条件
		- 仕様がわかっているので、その仕様を満たすかどうかテストしていく。
- 防御的テスト
	- 事前条件の範囲外もテストすること

欠陥は他の欠陥を隠す性質がある。
```
if 誤った条件:
	誤った処理
```
テストを実行することで、誤った条件に気づくことができるが
誤った処理には気付けない。
このような性質があるので、欠陥は一つずつ修正していく。

### ドメイン分析テスト、デシジョンテーブル
条件が複雑なときに使うのがこれらのテスト技法
複雑なのでテストケースの抜け漏れが発生しやすい

テストケースを考えるとき、図を書くとどういうテストを書けばいいのか、理解が深まる。

- ドメイン分析テスト
	- 2つの条件が相互作用を持つようなとき使われる設計技法
- 4つのポイント
	- on
		- 境界値
	- off
		- 境界値付近の値
		- onがドメイン内の値ならドメイン外にする
		- ドメイン外の値ならドメインないにする
		- 網羅したいので。
	- in
		- on, off以外のドメイン内の値
	- out
		- on., off以外のドメイン外の値
- 境界値テストはonポイント、offポイントのテストと言える。

- 複数条件を組み合わせて検証したい
	- 欠陥は他の欠陥を隠すので、検証箇所は一箇所にしないといけない。
	- 着目する一つの変数だけ固定する
		- 数学の二変数関数の最大最小問題のような感じ。
	- ドメインテストマトリクスを使ってテストケースを設計する。
- ドメインテストマトリクス
	- 一つの条件はinで固定する。残りの条件をon, offで検証する、というのを二つの条件について考える。

## わかったこと
 ### カバレッジ
- ホワイトボックステストの観点からテストケースを設計していく
- 今までのテスト技法は仕様を確認してテストケースを作っていた。すなわち、ブラックボックステストの観点から作っていた。
- ブラックボックステスト
	- コード内部のロジックを考慮しない
	- 外部から見た時の仕様
- ホワイトボックステスト
	- コード内部のロジックを考慮
	- 以下の二つに分類
		- データフローテスト
			- 変数のライフサイクルが守られているかどうか確認する
			- IDEが自動的にやってくれたり静的解析ツールがやってくれる。
		- 制御フローテスト
			- コード中の処理がどれくらい実行されたかのテスト
			- 100行のうち何%実行されたのかテスト
			- 全ての処理を実行させるにはどのようなテストケースがあればOKなのかを考える。
- カバレッジ
	- テスト可能なコードのうち、実際にテストされたコードの割合
	- テストの網羅性の程度として、カバレッジレベルがある。
- 5つのカバレッジレベル
	- ステートメントカバレッジ
	- デシジョンカバレッジ
	- コンディションカバレッジ
- ステートメントカバレッジ
	- 命令網羅、C0
	- 実行可能な行数のうち何行が実行できるのか？
	- 実行可能な行数とは空行、とじかっこ、コメント行など。
		- if文の実行も含む
- デシジョンカバレッジ
	- 判定網羅、分岐網羅、C1ともいう
	- 判定の分岐をどれだけ網羅しているか？
	- 判定の分岐とは
		- if文1個につきTrue/Falseの2通りの判定がある。
		- 2個ならば、4個の判定がある
		- そのうち、テストケースがどこまで網羅できるか？
	- より網羅性が高い
	- C0では100%でも50%になる。
	- if文が2つある場合、True, TrueとFalse, Falseになるテストケースを作れば100%になる
	- 全部のif文でTrue/Falseを見れるテストケースを作ればOK
	- if文のブロックを1単位で見る感じで、それぞれtrue, false確認できればOK
- コンディションカバレッジ
	- 判定の中の条件式の真偽をどれだけ網羅したか
	- C2、条件網羅ともいう
	- if文の中身の条件を1単位としてみる
	- 以下のコードならif文の中の条件の組み合わせは4通り、合わせて8通り
		- 判定の条件式のtrue/falseを全て確認できればOK
		- 条件式に限って注目する
			- 組み合わせで考えるのは次
			- 単体の条件のTrue/Falseがどれだけ現れたかで考える
```
if a > 0 and c == 1:
	pass
if b == 2 or d < 0:
	pass
```
- MCC
	- 複合条件網羅
	- 条件式の真偽の組み合わせで考える
	- メリット
		- 網羅性
	- デメリット
		- テストケースが増える
		- なので、MD/DCカバレッジが使われがち

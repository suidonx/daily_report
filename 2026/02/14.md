## わかったこと
- 再レンダリング
	- stateの更新が行われると、コンポーネントが再度読み込まれて、再レンダリングされる。
	- propsを受け取るコンポーネントのpropsの値が変わった時
	- 親コンポーネントが再レンダリングされた、子コンポーネントも再レンダリングされる。
- Strictモードかつ開発モードの時、2回レンダリングが走る
	- バグ検出のため
- useEffect
	- 第二引数の配列に渡したstateが更新されたときに、引数の関数が実行される
	- 処理の関心を切り分けることができる。
```
useEffect(() => {
	console.log("--useEffect--");
}, [num]);
```
- Reactが解決した課題
	- 従来の命令型UI
		- 命令を積み上げて組み立てる
		- 今、UIがどう構築されているか気にしながら作る必要がある
	- 宣言型UI
		- 状態を宣言して自動で構築
		- 今や過去の状態を気にしなくてもUIを構築してくれる
	- 高凝集・疎結合
		- コンポーネントにロジック（イベント）とUIを集める
		- 同じファイルに記述してあり、距離が近くて保守性が高い
		- 疎結合同士にすると再利用性が高い
- JSX
	- JavaScriptを拡張した構文
	- React.createElement()という関数に変換されて実行
- 仮想DOM
	- 変更前と変更後の仮想DOMのうち、差分だけを実体のDOMを更新している
	- 配列を使って作ったliタグにkeyを用意しておく
	- keyがないと、差分を取るために比較が走ると際、Reactは順番によって判断するため、li要素のうちどれを更新されたか？がわからない。
- onChange
	- 1文字ごとに発火するイベントハンドラ
- splice
	- 文字の削除、追加、置換ができる破壊的メソッド
- コンポーネント内でスタイルを定義
	- コンポーネントにスタイルを定義して外から変更されないようにする。

### React ステップアップ
- 再レンダリングの条件
	- stateが更新されたとき
	- propsが変更されたとき
	- 再レンダリングされたコンポーネント配下の子要素は再レンダリング

- 子コンポーネントをmemoで囲む
	- propsが変更されたとき以外は、再レンダリングしない

- アロー関数で書いた関数は、毎回新しく作られる
	- それがpropsに入ると、そのオブジェクトは前回のpropsの中身と、今回のpropsの中身とで比較して、メモリアドレスの参照先が違うという結果になる
	- そして、propsが変わったと判断し、再レンダリングされる

- useCallback
	- 関数をメモ化するフック
	- メモ化
		- 同じ参照を維持する
		- 同じものを使い回す

- CSSの当て方
	- インライン記法
		- オブジェクトの中にスタイルを書き、それをstyleタグに指定する
	- CSSモジュール
		- 拡張子が.module.scssといったcssファイルを書き、それをimportする
		- CSSをそのまま書くやり方
	- Styled JSX
		- cssファイルをstyleタグに直書きしていく形
		- next.jsにデフォルトで入っている
	- Styled components
	- Emotion

### Next.js完全攻略
- Pages Routerの時はページ単位のレンダリングで以下のレンダリング戦略があった
	- SSR, SSG, ISR, CSR
- App Routerはコンポーネント単位でレンダリングするので以下の呼び方になった
	- Dynamic Rendering
	- Static Rendering

- Server Component
	- App Routerのデフォルト
	- Dynamic Rendering
		- リクエストがあったときにサーバーでレンダリングする
	- Dynamic APIやライブラリ(Prisma)が使える
		- Server Componentでしか利用できない
		- 逆にuseState等は利用できない
	 - 非同期コンポーネントにできる
		 - Client Componentはできない
- Client Component
	- Rendering方式を決めるのはRSC
		- Dynamic APIを使うかどうかでDynamic Renderingか決まるから
		- Dynamic APIを使えるのはServer Componentなので。
	- Client ComponentはインタラクションなUII担当
		- Client ComponentでないとuseStateなど使えない
		- イベント処理、useState, useEffectが使えて、インタラクションを実現
		- use clientと宣言しないとClient Componentとみなさない
		- つけない場合、インタラクションの記述はエラーになる（Server Componentで使えないため）
	- インタラクションな処理を持つJSはハイドレーション

- Dynamic RenderingとStatic Renderingの判定
	- ユーザーごとやリクエストに応じて変わる処理を使う場合はDynamic、それ以外はStaticと判定される
	- デフォルトはStatic Rendering
	- Dynamic Rendering
		- cookies()、headers()、searchParams()といったAPIを使うとき
		- fetch関数にcache: "no-store"オプションを渡すとき
		- リクエストが起きたときにサーバー側でレンダリング
		- 内部の動き
			- JSXからJSをビルドして生成
			- リクエストがあるたびJSを実行してRSC Payloadとクライアント用のJSを生成
			- 次にHTMLを生成
	- Static Rendering
		- ビルド時にレンダリング
			- 必要なタイミングでHTMLを再構築
		- 基本はstaticにするという思想がNext.jsにある
			- ページ表示が高速
			- SEO対策
			- コンテンツがキャッシュされてサーバー負荷軽減
		- Dynamic APIを使わないこと
		- fetchにno-storeがない
		- 内部の動き
			- ビルド処理でRSC Payloadを作成、Client ComponentのJSXからJSを作成
			- 次に、RSC PayloadからHTMLを生成
			- クライアント側で、RSCPayloadとHTMLを受け取る
			- 必要なJSをダウンロードしてハイドレーション
			- 特定の関数が呼ばれたらRSC Payloadを再生成してHTMLを再構築
- RSC Payloadとは
	- 効率的なレンダリングをするために生成されるデータ構造
	- 1Pageにつき1つ生成
	- 内部にあるのは
		- HTMLのツリー構造、ネストしたClient Component, Server Component、そしてpropの内容
		- Client ComponentはJS
		- Server ComponentはHTML
	- Client Componentの場所がわかるので効率的にハイドレーションできる
	- JSX からRSC Payloadの再生成を行い、変更の差分を効率的に部分的な再レンダリング
			
- Streaming SSR
	- Dynamic RenderingにおいてHTMLを一度にブラウザへ送るのではなく、順次ブラウザへ送る
	- 通常は全てレンダリングし終わってページ全体のHTMLができたらブラウザへ送信される
		- ページの一部分だけ先に表示できるので表示速度が向上
	- ユーザインタラクティブな操作がなければ、Server Componentでデータ取得していきたい
		- コード量も少ない
		- Client Comopnentの場合、クライアント側でのレンダリングになる
		-  ブラウザがJSを読み込むまでUI操作可能にならない
			- Server Componentは非同期にできるのでUI操作可能

- Composition-pattern
	- Client Component以下はClient Componentとして認識される
		- なるべく末端のコンポーネントをClient Component化したい
	- use clientを宣言するところがClient boundary
		- なのでClient ComponentでServer Componentを利用すると、Client Componentとして認識されるためビルド時にエラーになる
			- 本来は展開されてHTMLにならなければいけないところが、JSになる
			- Composition-patternを使って解決できる
				- page.tsxにてClient ComponentのchildrenにServer Componentを渡すと、Server Componentを認識してくれるので正しくビルドされる

- キャッシュ
	- パフォーマンス向上とコスト削減のために可能な限りキャッシュを行う
	- Request Memoization
		- 1回のリクエストで同じURLをfetchしていたらキャッシュを見に行く、missの時、URLをキーに取得したdataをSETして、HITしたらキャッシュを使う
		- Server Componentでのfetch関数でこのような挙動になる
		- Client Componentでは起きない
	- Data Cache
		- 別ページ間で、fetch関数で同じURLを叩いている場合、初回はデータソースからファイルを取得し、ファイルに保存
		- 2回目はファイルを見に行って取得
		- fetch関数で"force-cache"をつける
		- ビルド後のfetch-cacheディレクトリに格納されている
	- Full Route Cache
		- Static Renderingで生成されたHTMLとRSC
	- Client Route Cache
		- ブラウザのメモリ上に訪問したページのRSCPayloadを保存
		- ソフトナビゲーション(戻る、進む)時に保存したRSCPayloadからHTMLを構築して表示
		- 5分でキャッシュ削除
		- LinkタグにセットしたURLをプリフェッチして、ブラウザのメモリ上にRSC Payloadを保存してキャッシュを作る

- ファイルシステム規約
	- 特定のファイルやディレクトリ構造が特別な意味や機能を持つ
	- layout.tsx
		- セクション全体の共通レイアウトを定義
		- app/のlayoutが外側のレイアウト
		- ディレクトリ配下のコンポーネントに対してレイアウトが適用
		- ディレクトリで別々のlayoutを当てるも可能
	- loading.tsx
		- ローディングUIを定義するファイル
		- layout.tsxは初期描画時に表示される
		- データ取得中などページが表示される待ち時間はpage.tsxではなくloading.tsxが表示
		- layout同様、ディレクトリで別々のloadingを当てることができる
	- error.tsx
		- global-error.tsxはapp/に置くことが多い
		- error.tsxは各pageのerrorUIを置く
	- route.ts
		- APIエンドポイントを定義するファイル
		- /api/usersというディレクトリにroute.tsを作成してjsonを配置する
	- middleware.ts
		- リクエストが処理される前に実行される中間処理を定義するファイル
- Dynamic Route Segments
	- 動的なルートパラメータを持つページを作成する機能
	- app/users/[id]/page.tsx
	- app/users/1でid=1を取得
- Parallel Route
	- 同じレイアウト内で複数のページを同時に表示する機能
	- ↑？
	- 例えば、@header/default.tsxをapp配下に作ると、同じセクション内のlayoutでpropsにheaderを受け取ることができる
- Intersepting Route
	- 別のルートをインターセプトできる機能
	- 例えば@modal/default.tsをapp配下に作ると同じセクション内のlayoutでpropsに渡せる
	- まだわからない
